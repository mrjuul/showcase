A Practical Guide for Humans

01 What Is Git?

Git is a time machine for folders. It tracks every change you save, lets you undo mistakes, and syncs your files to a server (like GitHub) so they're accessible from anywhere.

A folder becomes a git repo when you run git init inside it. This creates a hidden .git/ directory that stores the entire history. Without .git/, it's just a normal folder — no memory, no undo, no remote access.

📸

Snapshots, Not Diffs

Every commit is a full picture of all your files — not a list of changes

🔑

Content-Addressed

Every object gets a SHA-1 hash fingerprint — same content = same hash = stored once

🌍

Distributed

Every clone is a full copy with complete history — no server dependency

02 Why Your Files Are Everywhere

When you git push, your commits travel to GitHub. From there, anything with internet access can reach them — Claude.ai, your phone, another PC, a cloud sandbox.

Your PC ──git push──▶ GitHub ◀── Claude.ai / Mobile / Anywhere

Without git, your files would live only on the one machine they were created on.

03 Your Root Directory Is NOT a Git Repo

And that's correct. Your home folder contains system files, downloads, caches, and secrets. Making it a git repo would mean:

⚠ What happens if you git init in ~

git status shows thousands of irrelevant files. You risk accidentally committing credentials, browser data, or multi-gigabyte binaries. The repo becomes unusable noise.

Instead, each project is its own repo:

C:\Users\yourname\ ├── YourFileTree/ ← git repo (conventions, projects, config) ├── vault/ ← git repo (Obsidian knowledge base) ├── Documents/ ← not a repo (org documents) ├── Downloads/ ← not a repo (throwaway files) └── Desktop/ ← not a repo

Each repo is independently versioned and pushed to its own GitHub repository.

💡 Accidentally ran git init in ~?

Just delete the .git folder: rm -rf ~/.git — your files are untouched, only the empty git history is removed.

04 The Three Zones

Files in a git repo exist in three states. Understanding this is the key to everything.

┌─────────────────┐ ┌───────────────┐ ┌──────────────┐ │ Working │ git add │ Staging Area │ git commit │ Repository │ │ Directory │ ──────▶ │ (Index) │ ─────────▶ │ (.git/) │ │ │ │ │ │ │ │ Your actual │ │ What your │ │ Saved │ │ files on disk │ │ next commit │ │ history │ │ │ │ will contain │ │ forever │ └─────────────────┘ └───────────────┘ └──────────────┘

The staging area lets you commit some changes while keeping others as work in progress. This is why git add and git commit are separate steps.

05 Core Concepts

Commits = Save Points

Each commit is a snapshot of your entire project. Like a save point in a game — you can always load any previous one.

git add file.md            # stage a file for the next save
git commit -m "Add notes"  # save the snapshot

Branches = Parallel Timelines

A branch is a separate line of work. The default is main. Branches are just labels pointing to commits — they're free to create.

git branch experiment       # create a branch
git checkout experiment    # switch to it
# ... make changes, commit ...
git checkout main          # switch back
git merge experiment       # bring changes into main

SHA-1 Hashes = Fingerprints

Every object in git gets a unique 40-character ID based on its content:

"hello" → aaf4c61ddcc5e8a2dabede0f3b482cd9aea9434d

Change one byte and the hash is completely different. Same content anywhere in history = same hash = stored only once. This is how git is fast and efficient despite storing full snapshots. You see these as commit IDs like a3f8b21.

Remotes = Server Copy

A remote (usually origin) is your repo on GitHub. Push uploads. Pull downloads.

git push origin main       # upload commits to GitHub
git pull origin main       # download new commits

06 Essential Commands

Daily Use

git status                 # what's changed? what's staged?
git diff                   # see unstaged changes line by line
git add <file>             # stage specific file
git commit -m "message"   # save a snapshot
git push                   # upload to GitHub
git pull                   # download from GitHub

Seeing History

git log --oneline          # compact commit list
git log --oneline --graph  # visual branch diagram
git show <commit-id>       # what a specific commit changed
git blame <file>           # who changed each line and when

Undoing Things

git restore <file>         # discard uncommitted changes
git restore --staged <file> # unstage (keep the changes)
git revert <commit-id>     # undo a commit (safe for shared history)
git stash                  # shelve all changes temporarily
git stash pop              # bring them back

07 Safety Net: Reflog

Git records every action you take. Even after a terrible mistake, your work is recoverable for about 30 days:

git reflog                 # see every HEAD movement
git reset --hard <hash>    # go back to a specific point
💡 The golden rule of recovery

If you committed it, it's almost certainly still there. Check git reflog, find the hash, and reset back to it.

08 Things to Avoid

Don'tWhyDo Instead
git init in home dir Tracks system files, creates unusable noise Init inside project folders only
git add . blindly May stage secrets, binaries, system files git add <specific files>
Commit .env or API keys They live in history forever, even if deleted later Use .gitignore
git push --force Overwrites others' work on shared branches Regular git push
git reset --hard Destroys uncommitted work instantly git stash first

09 The .gitignore File

Tells git which files to never track. Put it in the root of your repo. Files listed here won't appear in git status and can't be accidentally committed.

# Secrets
.env
*.keys.json
credentials/

# OS junk
Thumbs.db
.DS_Store

# Dependencies
node_modules/

# Large files
*.zip
*.exe

10 Git + Obsidian Vault

An Obsidian vault is a folder of markdown files — a perfect fit for git.

vault/ ├── .git/ git tracks the vault ├── .gitignore exclude workspace settings ├── knowledge/ your notes ├── sessions/ session logs └── ...
Obsidian + Git Tips

Add .obsidian/workspace.json to .gitignore — it changes every time you open a pane.

Commit often. Each commit is a snapshot of your entire knowledge base.

Push to GitHub for backup and remote access (Claude.ai, mobile).

For large images/PDFs, consider git-lfs to keep the repo lean.

11 Mental Model

Folder + .git/ = time machine ├── commit = save point ├── branch = parallel timeline ├── push = upload to server (GitHub) └── pull = download from server No .git/ = just files on disk — no history, no remote access

Git is the reason your files show up on Claude.ai, on your phone, and survive mistakes. Keep it in your project folders, keep it out of your home directory, and commit often.
