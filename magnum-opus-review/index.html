01 — Strengths

What works

The foundation is solid. The modules are the best part of this project — genuinely portable, well-structured, and honest about gaps.

The Human Is the Constant
This isn't just a slogan. The modules are truly vendor-agnostic — paste 00-jonas-core.md into GPT, Gemini, or a local model and it works immediately. That's real portability, not marketing.
Module Structure
Each file is self-contained, ADHD-friendly (short, scannable), and maps to a clear domain. The domain-switch prompt is a practical pattern for context switching without infrastructure.
Honest Gap Tracking
06-gaps.md lists what's unknown. Most people skip this. Knowing your blind spots is underrated — it's the foundation for systematic improvement.
Clean Reorganization
Going from 66 scattered Google Drive files to 6 organized top-level folders is real progress. The mapping between modules and Drive folders is intentional and clear.
02 — Problems

What's missing

The modules describe who Jonas IS. They don't describe what the system is DOING. There's no state, no memory, no engine.

No Agent State
No SPEC.md, no changelog, no cross-session memory. Every Claude session starts from zero — reads modules, knows who Jonas is, but has no idea what was discussed yesterday. Session 2 can't continue Session 1's work.
Google Drive Lock-in
Despite "vendor-agnostic" principle, ADR-001 makes Google the truth layer. Claude Code can't natively read G:\My Drive — it needs API + OAuth. The modules already live in git. That's the more agentic pattern.
Wiki Without an Engine
The Karpathy wiki vision is solid. But /wiki-ingest, /wiki-query, /wiki-status don't exist. A wiki nobody maintains dies within weeks. The engine needs to be built before the knowledge graph.
Security Gaps
cv-oplysninger.pdf committed to git (personal data in version history). settings.local.json exposes full filesystem layout with 120+ path entries. No .gitignore at all.

04 — Roadmap

The path forward

Five concrete steps, ordered by impact and effort. Start with security, end with architecture.

Add .gitignore + remove CV from tracking
git rm --cached cv-oplysninger.pdf, add .claude/ and *.pdf to .gitignore. Stops leaking personal data and filesystem layout.
~30 min • fixes security
Add SPEC.md with changelog
A single file that tracks what's done, what's in progress, and what's next. Gives Claude cross-session state without building any infrastructure.
~1 hour • enables state tracking
Separate filing from agent state
Google Drive = filing cabinet for PDFs, contracts, shared Palmehaven docs. Git repos = agent state for decisions, project status, module evolution. Stop treating them as the same problem.
~2 hours • clarifies architecture
Build the wiki engine before the wiki
The /wiki-ingest and /wiki-status skills need to exist before the Karpathy wiki can work. Or — consider whether the modules ARE the knowledge layer already, and skip the wiki entirely for now.
~1-2 days • or skip it
Knowledge graph over modules
Graphify / Graphthulhu over existing modules might give you the visual overview and cross-domain connections you want — without maintaining wiki pages.
~1 day • alternative to wiki
05 — Bottom Line

The verdict

The modules are genuinely good. THE HUMAN IS THE CONSTANT is the right principle. But the system around the modules — state tracking, memory, automation, security — is where the real work begins.

Jonas is at the "I have context" stage. The next stage is "I have an agent that uses context to do work autonomously." That's a different engineering problem, and Google Drive won't solve it.

The good news: he's already using git without fully realizing it's the better foundation. The modules are portable. The reorganization is done. The path forward is about building the engine, not redesigning the context.
