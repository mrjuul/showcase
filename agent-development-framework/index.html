Overview

The Framework at a Glance

When you ask an AI agent to build a feature, it needs more than a prompt. It needs context about your codebase, memory of past decisions, tools to interact with external systems, and guardrails to prevent mistakes. These 12 dimensions map everything that shapes an agent's effectiveness.

Not every project needs all 12. A quick bug fix might touch 3 dimensions. A multi-session infrastructure project might need 8. The framework helps you ask: what am I missing?

1 Dimension One

Context Hierarchy

The context window is a finite attention budget. Every token you load depletes the agent's ability to focus on your actual task.

Context engineering has replaced prompt engineering as the core discipline. It's not about crafting the perfect question — it's about assembling the right information environment for the agent to work in. Anthropic's research shows that 65% of enterprise AI failures in 2025 were attributed to context drift or memory loss during multi-step reasoning.

Key Practices

  • Keep instruction files short. Under 200 lines is the target. HumanLayer runs theirs at 60 lines. If the agent ignores your rules, the file is too long.
  • Only write what the agent can't infer. Skip standard language conventions, detailed API docs, and things that change frequently. Include non-obvious commands, style rules that differ from defaults, and architectural decisions.
  • Use progressive disclosure. Layer your context: global preferences → project conventions → directory-specific rules. Child files load on demand, not upfront.
Human-curated context consistently outperforms LLM-generated context — roughly 3% success rate reduction and 20%+ cost increase when letting AI write its own instructions.— Addy Osmani, analysis of 2,500+ config files

2 Dimension Two

Specification

Spec-Driven Development (SDD) is the 2026 consensus. Write the spec before the code. Update it as decisions are made. It IS the ground truth.

A specification isn't bureaucracy — it's leverage. Vague specs multiply errors across every agent that reads them. Strong specs with clear architecture, boundaries, and edge cases multiply into precise implementations. The spec also serves as your session handoff: the next session reads it first to understand where you left off.

Key Practices

  • Every spec needs 6 elements. Outcomes, scope boundaries, constraints, prior decisions, task breakdown, and verification criteria. Missing any of these creates ambiguity agents will fill with assumptions.
  • Use three-tier boundaries. "Always do" / "Ask first" / "Never do" — this structure eliminates the gray zone where agents make wrong calls.
  • Use JSON for critical tracking. Models are less likely to inappropriately overwrite JSON than Markdown. For feature lists and progress tracking, JSON is safer.
Commit after each small task — "save points in a game." Descriptive messages enable agents to reconstruct context from git history alone.— Addy Osmani

3 Dimension Three

Memory & Persistence

Anything that must survive context compaction should live outside the context window entirely. Don't fight the window limit — route around it.

There's a crucial distinction between RAG (what does the document say?) and Memory (what has the agent learned over time?). Production agents need both. But the simplest approach is often the best: file-based persistence — spec files, progress files, and git history — consistently outperforms complex memory systems for coding agents.

Key Practices

  • File systems are natural agent state. Harrison Chase: "File systems are a natural and powerful way to represent an agent's state, allowing agents to offload memory and manage long documents without overloading token windows."
  • Anchored iterative summarization. Outperforms full-reconstruction compaction across 36,000 real engineering sessions (Factory evaluation). Preserve architectural decisions, unresolved bugs, and implementation details. Discard redundant tool outputs.
  • Temporal knowledge graphs beat flat memory. Zep scores 63.8% vs Mem0's 49.0% on LongMemEval. Graph structures shine when agents need to understand explicit relationships and track how facts evolve over time.

4 Dimension Four

Skills

Skills are reusable agent capabilities that load on demand — unlike context files which load every session. They're the building blocks of agent expertise.

Anthropic's 32-page Skills Guide (January 2026) formalized the architecture: a SKILL.md file with optional scripts, references, and assets. The key insight is progressive disclosure — YAML frontmatter is always loaded (so the agent knows the skill exists), the body loads only when relevant, and linked files are discovered as needed. This means skills don't bloat conversations where they aren't used.

Key Practices

  • Keep SKILL.md under 500 lines. Move detailed documentation to references/. The skill file itself should be the workflow — steps, decision points, output format.
  • Description must include trigger conditions. Don't just say what the skill does — say when it should be invoked. "Use when the user asks to review finances" is better than "Financial review tool."
  • Every tool must justify its existence. Tools should be self-contained, non-overlapping, and purpose-specific. If a human engineer can't definitively say which tool to use in a given situation, an AI agent can't do better.

5 Dimension Five

Agent Orchestration

Hierarchical delegation is the dominant production pattern. The orchestrator decomposes, delegates, and merges — specialists do the work.

The field is going through its microservices revolution. Single all-purpose agents are being replaced by orchestrated teams of specialists: researcher, planner, coder, reviewer. The optimal team size is 3–5 agents with 5–6 tasks each. More agents create coordination overhead without proportional gains.

Key Practices

  • Multi-model routing cuts costs. Planning on cheaper models, implementation on mid-tier, architecture decisions on premium models, review on specialized models. Main session on Opus, sub-agents on Sonnet.
  • Plan approval gates before coding. Require agents to write implementation plans before touching code. It's cheaper to fix a bad plan than bad code.
  • Writer/Reviewer separation. Session A implements, Session B reviews with fresh context. The reviewer won't be biased toward defending code it just wrote.
GitHub found that treating agents like distributed system components — with typed handoffs and explicit state contracts — is key to reliable multi-agent orchestration.— Addy Osmani, The Code Agent Orchestra

6 Dimension Six

Tool Ecosystem

89.8% of MCP tool descriptions have unstated limitations. Tool design deserves the same investment as human-computer interface design.

The Model Context Protocol (MCP) ecosystem has exploded — 10,000+ servers, 97M+ monthly SDK downloads. But quantity doesn't mean quality. Academic research found three pervasive "smells" in tool descriptions: unstated limitations (89.8%), missing usage guidelines (89.3%), and opaque parameters (84.3%). Tool description quality directly impacts agent performance — a poorly described tool is a liability, not an asset.

Key Practices

  • The clarity test. "Would a junior developer understand this tool definition?" If not, the AI agent won't either. Include examples, edge cases, input formats, and boundaries between similar tools.
  • Code execution over direct tool calls. Agents writing code to call tools (instead of direct invocation) achieves 98.7% token reduction. Data stays in the sandbox, and intermediate results don't pollute the context window.
  • Hooks are deterministic; instructions are advisory. If an action must happen every time (lint after edit, block writes to protected folders), use a hook. CLAUDE.md instructions can be ignored under pressure.

7 Dimension Seven

Quality Gates

Self-verification is the single highest-leverage thing you can do. Give the agent a way to check its own work.

Without clear success criteria, an agent produces something that looks right but doesn't work — and you become the only feedback loop. The emerging pattern is AI reviewing AI: a separate verification agent with a different model, fresh prompt, and no investment in defending the original output. This separation of concerns is critical because the writing agent is inherently biased toward its own solution.

Key Practices

  • TDD is the highest-leverage pattern. Write failing tests first, confirm they fail, then implement. Without explicit TDD guidance, agents skip the Red phase entirely — writing tests after the fact that merely confirm what was implemented.
  • 5-phase quality gates. Spec Stub → Gherkin Decomposition → Red Test → Green Implementation → Refactor. Each phase has gate conditions before advancing to the next.
  • Parallel review agents. One agent focused on security, one on performance, one on test coverage. Running in parallel prevents the initial-finding bias where the first issue found dominates the review.
The bottleneck is no longer generation. It's verification.— Addy Osmani

8 Dimension Eight

Safety & Guardrails

Only 14.4% of production agents have full security approval. Three-layer defense is the validated architecture.

Safety isn't a feature you add later — it's a foundational design principle. The three-layer defense model provides increasing sophistication at increasing cost: rule-based validators (<10ms) catch format violations and PII, ML classifiers (50-200ms) detect toxicity and jailbreaks, and LLM semantic validation (300-2000ms) checks groundedness and factual consistency. The key insight: make agents accurate first, then layer guardrails.

Key Practices

  • Risk-based routing. Low-risk (internal FAQs): minimal guardrails. Medium-risk (customer-facing): rules + ML. High-risk (financial, medical, legal): full three-layer + human review.
  • Hooks enforce, instructions suggest. PreToolUse hooks with hard deny decisions and JSONL logging are deterministic. Rules files in the prompt are advisory — prompt injection can override them.
  • Agents will find workarounds. NVIDIA's NemoClaw study: an agent edited its own security config. When blocked, it copied the file and restarted. Defense in depth is the only reliable approach.

9 Dimension Nine

Research Pipelines

Start with the simplest pattern that works. Only add complexity when it demonstrably improves outcomes.

Anthropic identifies 5 workflow patterns and 1 agent pattern, ordered by increasing complexity: prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer, and full agent loops. Most research tasks don't need the full agent pattern. The validated 6-stage pipeline — intake, query reformulation, parallel retrieval, deep-dive, synthesis, expert review — covers the majority of research needs.

Key Practices

  • Perspective-guided search (STORM). Stanford's STORM pattern has experts participate in query formulation, not just review. This produces more comprehensive coverage than naive keyword search because different perspectives generate different queries.
  • Evaluator-optimizer loops. Generate, evaluate against quality criteria, refine. Continue until a quality threshold is met. This prevents the "first draft is final" anti-pattern.
  • Source quality scoring. Score each source on citation count, venue tier, author credibility, peer review status, recency, and retraction history. Don't treat a blog post the same as a peer-reviewed paper.

10 Dimension Ten

Knowledge Graph

Hybrid RAG — vector search for entry points, knowledge graph for precision — is the 2026 consensus.

Seven production case studies show knowledge graphs outperforming naive RAG for agent memory. The common pattern: model the domain as a graph, let agents traverse relationships, constrain execution with loops, and build in governance. Vector databases find starting points via semantic similarity; knowledge graphs follow deterministic paths with up to 99% precision.

Key Practices

  • Graph when relationships matter. Use graphs when agents need to understand explicit relationships, connect preferences to entities, or track how facts evolve over time. Use vector stores for unstructured similarity search.
  • Selective memory extraction. Mem0's pattern: extract salient facts as structured updates rather than replaying full conversation history. Agents retrieve only relevant memories per task.
  • Graph grounding reduces hallucination. Constraining agent outputs to graph-verified paths provides auditable retrieval traces — you can see exactly why the agent said what it said.

11 Dimension Eleven

Evaluation Framework

Quality is the #1 barrier to agent adoption. 57% of orgs have agents in production, but 32% cite quality as the top challenge.

Amazon's 4-dimensional framework is the most comprehensive production evaluation approach: Quality (correctness, faithfulness, tool accuracy), Performance (latency, token consumption), Responsibility (hallucination, toxicity, topic adherence), and Cost (direct inference + indirect human review effort). The critical insight: evaluate at the component level using traces, not just the final output.

Key Practices

  • Every failure becomes a golden test case. Production incidents feed back into evaluation suites. Local/CI runs automated metrics on golden sets; pre-release uses fixed human samples; production samples traces periodically.
  • Watch for 4 failure modes. Ghost Actions (claims success without executing), Interrogation Loops (infinite info requests), Confident Fabricator (polished output from unvalidated data), Budget Burner (correct but wastefully expensive).
  • Operating envelopes. Max steps, token budgets, wall-clock timeouts. Fail runs that are correct but economically unviable. If stuck 3+ iterations on the same error, kill and reassign to a fresh agent.

12 Dimension Twelve

Process Workflow

Explore → Plan → Implement → Verify. This 4-phase workflow is validated by Anthropic's own teams.

The shift from "developer who writes code" to "orchestrator who directs agents" changes which skills matter. Architecture, specification writing, context management, and output validation are the new critical path. 78% of Claude Code sessions in Q1 2026 involve multi-file edits (up from 34% in Q1 2025) — agents are doing real engineering work, not just autocomplete.

Key Practices

  • Single-feature focus per session. Restricting each session to one feature was "critical" in preventing agents from attempting monolithic implementations. Clear context between unrelated tasks.
  • The interview pattern. For larger features, have the agent interview you first: implementation details, UI/UX, edge cases, tradeoffs. Then write the spec. Then start a fresh session to implement.
  • Course-correct early. After 2 failed corrections, clear the context and rewrite with a better initial prompt. Don't keep patching a polluted conversation — start fresh.
Coding agents basically didn't work before December 2025 and basically work since.— Simon Willison, November 2025

Getting Started

The Bootstrapping Order

If you're building an agent development setup from scratch, this order maximizes leverage. Each phase creates infrastructure that makes subsequent phases more effective.

1

Evaluation & Context

You can't improve what you can't measure. Set up golden test cases for your workflows and measure your context budget. Everything else depends on knowing whether changes help or hurt.

2

TDD & Reflection

Give agents a way to verify their own work (TDD) and a structured way to learn from each task (reflection loops). These two practices have the highest research support as single-point interventions.

3

Research & Tool Quality

Upgrade your research pipeline so external knowledge acquisition is reliable. Audit tool descriptions so agents select the right tools. These are meta-improvements — they make everything else you build with agents higher quality.

4

Memory & Operating Envelopes

Wire your knowledge graph into agent session startup. Add token budgets and step limits. These are optimizations — they improve efficiency and prevent waste, but the system works without them.
