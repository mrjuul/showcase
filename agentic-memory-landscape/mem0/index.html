Mem0

The most widely adopted open-source agentic memory layer

Architecture

Mem0 processes every conversation turn through a two-phase pipeline that decides what to remember, then stores memories across three complementary backends for retrieval at inference time.

Memory Scopes

Mem0 supports three scoping levels, each keyed by a different identifier. All scopes can coexist and are queried in combination at retrieval time.

Scope Lifetime Key Use Case
User Cross-session, persistent user_id Long-term preferences, facts, and history tied to a specific user across all conversations
Session Single conversation run_id Short-term context within one conversation thread, discarded or archived after session ends
Agent Per agent instance agent_id Agent-specific learned behaviors, tool preferences, and operational patterns

Graph Memory

Beyond vector similarity, Mem0 extracts entities and relationships from conversations using an LLM, then stores them as a knowledge graph. This enables multi-hop reasoning and relationship-aware retrieval that flat vector search cannot provide.

Supported Backends

Graph Databases

Neo4j, Memgraph, Amazon Neptune, Kuzu, and Apache AGE are all supported as graph store backends. The graph layer runs alongside the vector store, not as a replacement.

Benchmark

Mem0g Performance

Mem0 with graph memory (Mem0g) scores 68.4% on LOCOMO, compared to the base vector-only Mem0 at 66.9%. The graph layer adds relational reasoning at modest cost.

Extraction

Entity & Relation Pipeline

The LLM identifies named entities and typed relationships from each turn. These are deduplicated against the existing graph before insertion, maintaining a clean knowledge structure.

Key Patterns

Two architectural patterns distinguish Mem0 from simpler memory implementations.

Pattern

LLM-Driven Deduplication

Each candidate memory is compared against existing memories via vector similarity. An LLM then classifies the action as ADD, UPDATE, DELETE, or NOOP. This prevents unbounded memory growth and keeps the store semantically clean without manual curation.

Pattern

Actor-Aware Memory

The Group-Chat v2 update (June 2025) introduced actor-aware memory extraction. In multi-agent or multi-user conversations, Mem0 tracks which actor said what, enabling correct attribution and per-participant memory scoping within a shared conversation context.

Memory API

Mem0 exposes six core methods. All accept scope parameters (user_id, agent_id, run_id) for filtering.

# Initialize the memory client from mem0 import Memory m = Memory() # Add memories from a conversation m.add( "I prefer dark-roast coffee and work from Copenhagen", user_id="christian" ) # Search memories by semantic similarity results = m.search( "What does the user like to drink?", user_id="christian" ) # Update an existing memory by ID m.update( memory_id="mem_abc123", data="Now prefers light-roast coffee" ) # Delete a specific memory m.delete(memory_id="mem_abc123") # Get all memories for a user all_mems = m.get_all(user_id="christian") # View mutation history for a memory changes = m.history(memory_id="mem_abc123")

Accuracy Trade-off

~6% Gap

Mem0 scores 66.9% vs full-context 72.9% on LOCOMO. The graph-enhanced variant (Mem0g) narrows this to 4.5 percentage points.

Latency Reduction

91% Lower

By retrieving only relevant memories instead of passing the full conversation history, Mem0 achieves 91% lower latency at inference time.
