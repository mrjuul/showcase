MemGPT / Letta

OS-inspired virtual context management for LLM agents

01The OS Analogy

MemGPT treats the LLM context window like an operating system treats physical memory. Just as virtual memory lets programs use more RAM than physically exists, MemGPT lets agents use more context than the model window allows.

Operating System MemGPT Equivalent
Main memory / RAM Main context (the active token window)
Disk storage External context (recall + archival databases)
Virtual memory Virtual context (illusion of unbounded memory)
Page faults Heartbeat requests (self-triggered continuation)
Memory Management Unit (MMU) The LLM itself (decides what to page in/out)

02Tiered Storage Architecture

Context is split into two tiers with distinct access characteristics. The LLM manages data movement between them using dedicated memory tools.

Main Context (In-Window)
System Instructions read-only
Base persona, instructions, and tool definitions.
~1,076 tokens
Core Memory: Human writable
Persistent facts about the user. The agent edits this block in-place as it learns.
Core Memory: Persona writable
The agent's self-description and behavioral guidelines. Also self-editable.
Conversation History FIFO + summarization
Recent messages kept in a FIFO queue. When the window fills, oldest messages are recursively summarized before eviction.
External Context (Out-of-Window)
Recall Storage text search
Full conversation history stored as searchable text. Queried via conversation_search with date-range and keyword filters.
Archival Storage embedding search
Unbounded external memory. Data is embedded and retrieved via semantic similarity search. The agent can insert, search, and delete entries autonomously.

03Self-Editing Memory Tools

The agent is given a set of function tools to manage its own memory. Critically, send_message is the only tool that produces user-visible output — all other operations are internal reasoning.

Tool Target Description
core_memory_append Core Memory Append new information to the human or persona block
core_memory_replace Core Memory Edit existing text in-place (find-and-replace within a block)
archival_memory_insert Archival Storage Insert a new entry into long-term archival storage
archival_memory_search Archival Storage Semantic search over archived entries by embedding similarity
conversation_search Recall Storage Search past conversation history by keyword and date range
send_message User Send a message to the user — the only user-visible output
request_heartbeat System Request another reasoning step (self-triggered continuation)

04MemGPT vs Traditional RAG

Traditional RAG retrieves documents once per query. MemGPT gives the LLM continuous read-write access to its own memory, fundamentally changing the data flow.

Dimension Traditional RAG MemGPT
Data flow Unidirectional — retriever feeds context to LLM Bidirectional — LLM reads and writes its own context
Who manages context External pipeline (chunking, embedding, retrieval) The LLM itself via function calls
Memory updates External ETL process, batch re-indexing Real-time self-edits during conversation
Cross-session learning Does not learn from interactions Accumulates knowledge across conversations

05Key Patterns

Virtual Context Management

The LLM acts as its own memory controller, deciding when to page data in and out of the fixed-size context window. This creates the illusion of unbounded memory within a bounded token budget.

Self-Editing Core Memory

Instead of relying on external systems to update user profiles, the agent directly modifies its own core memory blocks using core_memory_append and core_memory_replace during conversation.

Recursive Summarization on Eviction

When conversation history overflows, messages are not simply truncated. Instead, they are recursively summarized — preserving semantic content while freeing token space for new interactions.
