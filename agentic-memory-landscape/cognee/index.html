Cognee

Self-improving knowledge graph memory engine

Three-Layer Storage

Every piece of data lives simultaneously across three complementary storage backends. Each graph node has a corresponding vector embedding, and relational tables track metadata, permissions, and lineage.

◈
Graph Layer
KuzuDB · Neo4j · FalkorDB
Entities, relationships, and ontological structure. Enables multi-hop traversal, Cypher queries, and graph completion. The backbone of Cognee's knowledge representation.
| every node has a corresponding embedding |
◖
Vector Layer
LanceDB · Qdrant · pgvector
Dense embeddings for semantic similarity search. Indexes chunks, summaries, and entity descriptions. Enables fuzzy matching and nearest-neighbor retrieval across the full corpus.
| metadata, permissions, and lineage |
▦
Relational Layer
SQLite · PostgreSQL
Structured metadata, access permissions, data lineage, and pipeline state. Provides ACID guarantees and serves as the coordination layer between graph and vector stores.

The Pipeline

Cognee processes data through four composable stages. Raw data enters via add(), gets structured by cognify(), evolves through memify(), and surfaces via search().

add()
Ingest 38+ data types: PDF, DOCX, PPTX, audio, video, images, Slack exports, code repos, URLs, raw text, and more. Automatic format detection and normalization.
→
cognify()
Six-stage processing pipeline:
  • 1 classify document type
  • 2 assign permissions
  • 3 chunk into segments
  • 4 extract entities + relations
  • 5 generate summaries
  • 6 compute embeddings
→
memify()
Self-improvement loop that evolves memory over time:
  • × prune stale facts
  • ↑ strengthen frequent
  • ↺ reweight from feedback
  • + derive new facts
→
search()
14 retrieval modes across graph, vector, and hybrid strategies. From classical RAG to graph completion to auto-routed "feeling lucky" mode.

memify() Self-Improvement

Most memory systems are static stores: data goes in, data comes out, nothing changes in between. Cognee's memify() is the key differentiator -- it treats memory as a living system that evolves.

Memory is not a database. It is a process.

After cognify() builds the initial knowledge graph, memify() runs continuously or on-demand to reshape what the system knows. Facts that are retrieved often get strengthened. Facts that contradict new evidence get pruned. User feedback reweights edges. And the system derives new composite facts by traversing its own graph -- knowledge that was never explicitly ingested.

✂
Prune
Remove stale, contradicted, or low-confidence facts. Keeps the graph clean and authoritative.
↑
Strengthen
Boost edge weights on frequently accessed paths. Popular knowledge surfaces faster.
↻
Reweight
Incorporate explicit user feedback to adjust relationship confidence scores.
✦
Derive
Traverse existing edges to infer new composite facts never explicitly ingested.

Key Patterns

memify Loop

Schedule memify() on a cron or trigger it after each conversation. The knowledge graph continuously self-improves: pruning contradictions, strengthening high-value paths, and deriving new relationships from existing data. No manual curation required.

Hybrid Vector + Graph

Vector search finds semantically similar content. Graph traversal follows structural relationships. Cognee combines both in a single query, re-ranking results by a weighted fusion score. This catches associations that pure embedding search misses.

Fully Local Deployment

Run entirely on your machine with zero cloud dependencies. Use Ollama for embeddings and LLM inference, KuzuDB for the graph layer, LanceDB for vectors, and SQLite for metadata. Privacy-first, air-gapped capable, no API keys needed.
