Agentic Memory Landscape

Chip Huyen

The production engineering lens on agent memory

01 Three-Tier Memory Framework

Huyen's contribution to the agent memory discourse comes from an engineering-first perspective. Where Weng maps cognitive psychology and Ng defines architectural patterns, Huyen asks the production question: how do you actually build and operate memory systems at scale? Her three-tier framework, presented in AI Engineering (O'Reilly, 2025), organizes memory by the system's relationship to the knowledge -- whether it is baked-in, session-scoped, or externally persisted.

Tier Description Implementation Scope
Internal Knowledge Knowledge encoded in model weights during pretraining and fine-tuning. The model "knows" this without any retrieval step. Pretraining data, RLHF, supervised fine-tuning, continued pretraining on domain data. Updated only through retraining -- expensive and infrequent. Static
Short-Term Memory Information available within the current context window. Includes the system prompt, conversation history, and any retrieved documents injected into the prompt. Context window management, prompt engineering, in-context learning. Limited by token budget -- typically 4K-200K tokens depending on model. Session
Long-Term Memory Persistent external storage that the model reads from and writes to across sessions. This is the tier where RAG, vector databases, and structured knowledge stores live. Vector databases (Pinecone, Chroma, Weaviate), relational databases, document stores, knowledge graphs. Requires indexing, retrieval, and update pipelines. Persistent

The engineering insight is that each tier has different update mechanisms, latency profiles, and failure modes. Internal knowledge is high-quality but frozen. Short-term memory is flexible but ephemeral. Long-term memory is persistent but requires retrieval infrastructure that can fail, return irrelevant results, or introduce stale information. Production systems must handle all three tiers and their interactions gracefully.

02 RAG vs Fine-Tuning Decision Matrix

One of Huyen's most practical contributions is a clear decision framework for choosing between the four main approaches to giving an LLM access to new knowledge. The choice depends on how dynamic the knowledge is, how much control you need over outputs, and what infrastructure you can afford to operate. There is no single best approach -- production systems often combine multiple options.

Option Best For Latency Update Speed Tradeoff
RAG Large, frequently-updated knowledge bases. Documents that change daily or weekly. When you need citations and source attribution. Higher -- retrieval adds 50-200ms per query. Embedding + search + reranking pipeline. Minutes to hours. Re-index changed documents without touching the model. Retrieval quality is the bottleneck. Bad retrieval means bad answers regardless of model quality. Requires vector DB infra.
Fine-Tuning Specialized domains, consistent output formatting, behavioral alignment. When the knowledge is stable and the model needs to "act differently" not just "know more." No retrieval overhead -- knowledge is in the weights. Inference latency unchanged. Days to weeks. Requires dataset curation, training run, evaluation, deployment. Catastrophic forgetting risk. Expensive to iterate. Hard to debug -- knowledge is opaque in weights. Cannot cite sources.
Long Context Small-to-medium knowledge bases (under ~500 pages). When retrieval quality is too unreliable. When you want the model to "see everything." Higher first-token latency due to processing full context. But no retrieval pipeline needed. Instant -- just swap the context payload. No indexing or training required. Cost scales linearly with context size. "Lost in the middle" attention degradation. Token limits still apply. Expensive at scale.
Prompting Small, critical information. User preferences, formatting rules, a few key facts. When the knowledge fits comfortably in the system prompt. Minimal overhead -- just a longer prompt. No infrastructure needed beyond the LLM. Instant. Edit the prompt template and redeploy. Does not scale. Token budget competes with conversation history. Easy to get wrong -- prompt engineering is empirical and fragile.

Huyen's key insight: RAG and fine-tuning are not competing approaches but complementary. Fine-tuning teaches the model how to behave; RAG gives it access to current information. The best production systems use fine-tuning for style, format, and domain behavior, then RAG for knowledge that changes. Prompting handles the handful of critical rules that must never be missed. Long context is the fallback when retrieval quality is unacceptable.

03 Memory Management Strategies

Context windows are finite. As conversations grow and memory accumulates, something must give. Huyen identifies three canonical strategies for managing memory within the constraint of a fixed token budget. Each makes different tradeoffs between completeness, relevance, and computational cost.

FIFO Eviction

Simplest

First In, First Out. When the context window fills up, drop the oldest messages. The system prompt and most recent turns are always kept. This is the default strategy in most chat applications and requires zero additional infrastructure.

if tokens > limit:
  drop oldest messages
  keep system prompt
  keep last N turns
Tradeoff: Loses early context that may contain critical setup information, user preferences, or decisions made at the start of a session. Works well for short, transactional conversations. Breaks down for long sessions with important early context.

Redundancy Removal

Deduplication

Detect and remove redundant information across the conversation. If the same fact appears in three different messages, keep only the most recent or most authoritative version. Requires semantic similarity comparison between memory entries.

for each new memory:
  find similar existing
  if sim > threshold:
    merge or replace
Tradeoff: More compute per write (similarity comparison is O(n) against existing memories). Risk of incorrectly merging memories that are similar but meaningfully different. Requires tuning the similarity threshold -- too aggressive loses nuance, too conservative wastes tokens on duplicates.

Reflection-Based

Most Sophisticated

Periodically use the LLM itself to summarize, consolidate, and abstract clusters of memories into higher-level insights. Raw memories are replaced by their distilled summaries, freeing token budget while preserving semantic content. This mirrors the Generative Agents reflection mechanism.

every K interactions:
  cluster recent memories
  LLM summarizes cluster
  replace raw with summary
Tradeoff: Highest quality compression but most expensive -- each reflection consumes LLM tokens. Summaries lose granular detail that may be needed later. Summarization can introduce hallucinations or drift from the original meaning. Recursive summarization compounds these errors.

Huyen emphasizes that production systems rarely use a single strategy in isolation. The practical approach is layered: FIFO as the baseline, redundancy removal for the working set, and reflection-based consolidation for long-term storage. The key metric is not how much the agent remembers, but how effectively it retrieves relevant memories at decision time.

04 Production Engineering

Caching as Memory Infrastructure

Huyen reframes caching -- traditionally an infrastructure optimization -- as a form of agent memory. When an LLM generates a response, caching that response means the system "remembers" the answer for identical or similar future queries. This is not semantic memory in the cognitive science sense, but it is functionally equivalent for production systems: the system avoids recomputing what it already knows.

Three Caching Layers

Prompt Cache
Cache KV-pairs for static prompt prefixes. Saves compute on system prompt processing. Provider-level (Anthropic, OpenAI).
Semantic Cache
Cache responses by query embedding similarity. Near-duplicate questions return cached answers. Application-level (GPTCache, Redis).
Result Cache
Cache exact query-response pairs. Deterministic lookup for repeated identical requests. Cheapest and most reliable tier.

The engineering challenge is cache invalidation -- the classic hard problem. When should a cached response be evicted? If the underlying knowledge changes, stale caches serve wrong answers with high confidence. Huyen recommends TTL-based expiration with semantic invalidation triggers: when a source document changes, invalidate all cache entries whose retrieval included that document.

Latency Metrics That Matter

Huyen defines three latency metrics that production teams must track for LLM-based systems. Memory operations (retrieval, caching, embedding) affect all three. A memory system that adds 200ms of retrieval latency might be acceptable for a research assistant but fatal for a real-time coding copilot.

TTFT
Time to First Token
How long the user waits before seeing any output. Dominated by prompt processing and retrieval latency. Memory retrieval adds directly to TTFT. Prompt caching reduces it. Target: under 1 second for interactive use.
TPOT
Time Per Output Token
Throughput during generation. Affected by model size, batch size, and hardware. Memory operations do not directly affect TPOT since retrieval happens before generation. But larger retrieved contexts increase KV-cache pressure.
TBT
Time Between Tokens
Inter-token latency during streaming. Jitter here causes stuttering in real-time applications. Memory writes (saving conversation state) should be asynchronous to avoid blocking generation. Target: under 50ms for smooth streaming.

Reliability Arithmetic

Huyen's most sobering contribution is a simple mathematical observation about compound reliability in multi-step agent systems. If each step in an agentic pipeline has independent reliability p, then a chain of n steps has compound reliability pn. This is multiplicative, not additive -- and the numbers get ugly fast.

The Compound Reliability Problem

0.9510
= 59.9%
Ten steps at 95% each yields ~60% end-to-end reliability

A single LLM call at 95% reliability is excellent. But an agentic workflow with 10 sequential LLM calls -- retrieval, planning, tool selection, execution, reflection, summarization -- drops to 60% end-to-end reliability. This is not a theoretical concern; it explains why complex agent demos work impressively but fail unpredictably in production.

3 steps @ 95%
85.7%
5 steps @ 95%
77.4%
10 steps @ 95%
59.9%

The implications for memory are direct: every memory retrieval step, every embedding lookup, every cache check is another link in the chain. Memory systems that add reliability (by catching errors or providing grounding) must add more reliability than they subtract through additional failure points. Huyen's prescription: minimize chain length, maximize per-step reliability, and build fallback paths that skip memory when retrieval fails rather than crashing the entire pipeline.
