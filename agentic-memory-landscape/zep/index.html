Bi-Temporal Model

Graphiti uses a bi-temporal data model where every fact carries four timestamps. This separates when something was known from when something was true, enabling the system to reason about knowledge evolution without losing history.

Timestamp Dimension Purpose
created_at System time When the fact was first recorded in the graph
valid_at Real-world time When the fact became true in the real world
invalid_at Real-world time When the fact stopped being true (set when superseded)
expired_at System time When the system learned the fact was no longer valid

Facts are never deleted. When a fact becomes outdated, it receives an invalid_at timestamp and a new edge is created with the updated information. This preserves the full evolution of knowledge and enables point-in-time queries.

Memory Concepts

Entity Nodes

People, places, objects, and concepts extracted from conversations. Deduplicated across episodes via embedding similarity.

Fact Edges

Temporal triplets (subject-predicate-object) connecting entities. Each edge carries bi-temporal timestamps and a natural language summary.

Episodes

Raw conversational data preserved as-is. Source material for entity and fact extraction. Linked to the entities and facts they produced.

User Graphs

Per-user knowledge graphs isolating individual memory. Each user's entities and facts are stored in their own graph namespace.

Group Graphs

Shared knowledge across user groups. Team or organizational memory that multiple agents can read from and write to.

Custom Entity Types

Developer-defined entity schemas with typed attributes. Constrain extraction to domain-specific node types (e.g., Patient, Product).

Fact Ratings

Confidence scores on extracted facts. Ratings can be updated as new evidence arrives, influencing retrieval ranking.

Community Detection

Graph clustering to identify related entity groups. Used for contextual retrieval and building coherent memory summaries.

Key Patterns

Bi-Temporal Fact Evolution

When new information contradicts an existing fact, the old edge receives an invalid_at timestamp and a new edge is created with updated content and a fresh valid_at. Both edges persist in the graph. This means an agent can answer "what did we know then?" and "what do we know now?" against the same graph, without data loss or destructive updates.

Hybrid Retrieval Pipeline

Queries are processed through three parallel retrieval paths: semantic search over entity and fact embeddings, BM25 keyword matching for precise term lookups, and graph traversal for structural relationships. Results are fused and re-ranked. Zep Cloud reports <200ms P95 latency for the combined retrieval pipeline, making it suitable for real-time conversational agents.

Incremental Graph Building

Each new episode triggers extraction, entity resolution, and edge creation without rebuilding the entire graph. Deduplication uses embedding cosine similarity with configurable thresholds. New entities are matched against existing nodes before creation, preventing graph bloat while preserving distinct concepts.

Point-in-Time Queries

The temporal model supports queries scoped to specific time windows. Retrieve the state of knowledge as of any past date by filtering on valid_at and invalid_at ranges. Useful for compliance, debugging agent behavior, and understanding how decisions were made based on the knowledge available at the time.
