Agentic Memory Landscape

LangChain Memory

From 8 deprecated classes to memory-as-filesystem

01Evolution Timeline

LangChain's memory story is one of aggressive deprecation. What began as tightly-coupled Python classes evolved into a decoupled, agent-owned filesystem model.

2023 — v0.x Era
8 Memory Classes Ship
ConversationBufferMemory, ConversationSummaryMemory, EntityMemory, and five more. Each hardwires a strategy into the chain. Works for demos, breaks at scale.
Sep 2024 — v0.3
Mass Deprecation
All 8 legacy memory classes deprecated. LangChain tells users to manage chat history themselves via RunnableWithMessageHistory or LangGraph checkpointers.
Dec 2024
Semantic Search Integration
Vector-store backed retrieval promoted as the long-term memory pattern. InMemoryStore and BaseStore become the endorsed primitives.
Feb 2025
LangMem SDK Launches
Dedicated library for agent memory: semantic, procedural, and episodic types. Introduces create_memory_manager, create_memory_store_manager, and namespace-scoped profiles.
Jul 2025
Deep Agents & Persistent Memory
LangGraph "deep agent" patterns formalize long-running agents with persistent checkpoints, shared memory namespaces, and cross-agent knowledge transfer.
2026
Agent Builder & Platform Memory
LangSmith Agent Builder adds hosted memory stores. Memory becomes a platform feature, not a library concern. Users configure memory via UI, not code.
Apr 2026 — Present
"Your Harness, Your Memory"
Community converges on filesystem-as-memory: AGENTS.md for procedural knowledge, tool configs for semantic facts, and harness-native checkpointers for state. The framework recedes.

02The 8 Deprecated Classes

Every one of these was deprecated in LangChain v0.3 (September 2024). They remain importable from langchain_community but emit deprecation warnings and receive no fixes.

Class What It Did Replacement
ConversationBufferMemory Stored entire conversation verbatim LangGraph checkpointer + message trimming
ConversationBufferWindowMemory Kept last k message turns LangGraph checkpointer + window filter
ConversationSummaryMemory LLM-generated running summary LangGraph summarization node
ConversationSummaryBufferMemory Hybrid: recent buffer + older summary Custom LangGraph reducer
ConversationTokenBufferMemory Kept messages up to token limit trim_messages() utility
ConversationEntityMemory Extracted & tracked named entities LangMem semantic memory + BaseStore
ConversationKGMemory Built knowledge graph from conversation LangMem + external graph DB
VectorStoreRetrieverMemory Semantic search over past messages InMemoryStore / BaseStore + embeddings

The core lesson: memory strategies were too diverse to bake into a single class hierarchy. LangChain chose composability over convenience.

03Current Architecture

Post-deprecation, LangChain splits memory into two clean layers, both built on LangGraph primitives.

Short-Term

Checkpointers

LangGraph's MemorySaver, SqliteSaver, or PostgresSaver persist the full graph state after each step. Thread-scoped by default. Replay, branch, and time-travel are built in.

This replaces all the buffer/window/token memory classes. The agent framework manages message history; you just pick a storage backend.

Long-Term

BaseStore + LangMem

BaseStore provides a namespace-scoped key-value interface. InMemoryStore adds vector search with configurable embeddings. LangMem layers cognitive memory types on top.

Cross-thread by design. Agents write memories during execution and read them on subsequent runs. No conversation coupling.

Bridging Layer

Memory Injection Pattern

At graph entry, a "memory node" queries BaseStore for relevant memories and prepends them to the system prompt. At graph exit, an "extraction node" pulls new facts from the conversation and writes them back. This is the canonical pattern in LangGraph documentation as of 2026.

04LangMem SDK

LangMem is LangChain's dedicated memory library. It models three cognitive memory types and provides APIs for each.

Type Definition API Update Pattern
Semantic Facts, entities, relationships. "User prefers dark mode." World knowledge distilled from interactions. create_memory_store_manager Upsert: new facts merge with or replace existing entries. Conflict resolution via LLM.
Procedural How to do things. Instructions, style guides, tool-use patterns. "Always respond in bullet points." create_memory_manager Prompt patching: the LLM rewrites its own system prompt based on learned procedures.
Episodic Specific past events. "On Tuesday the deploy failed due to OOM." Few-shot examples from history. create_memory_store_manager Append-only: episodes accumulate, retrieved by similarity. No merging.
Namespaces

Multi-Tenant Scoping

All LangMem operations are scoped by namespace tuples like ("user", "alice") or ("org", "acme", "project", "alpha"). This allows a single store to serve multiple users, teams, or agents without cross-contamination. Namespaces compose naturally with LangGraph's thread model.

05Memory as Filesystem

The latest evolution moves memory out of databases and into version-controlled files. The agent's working directory is its memory.

.agent/
  AGENTS.md ← procedural memory (how to behave)
  skills/
    code-review.md ← semantic memory (domain knowledge)
    deploy-pipeline.md
  tools.json ← config memory (available capabilities)
  memory/
    user-preferences.json ← semantic (entity facts)
    past-incidents.jsonl ← episodic (append-only log)
Procedural

AGENTS.md / CLAUDE.md

System-prompt-level instructions that shape agent behavior. Version-controlled, human-editable, loaded every session. The agent can propose edits; a human or CI gate approves them.

Semantic

Skills & Knowledge Files

Markdown files containing domain expertise, reusable procedures, and reference data. Loaded on demand via file reads or tool dispatch. Searchable, composable, diffable.

Config

tools.json & Settings

Declarative configuration defining what tools, MCP servers, and integrations the agent can access. Memory of capability rather than content. The agent's "hands."

This pattern is framework-agnostic. Whether you use LangGraph, CrewAI, Claude Code, or a bare API loop, the filesystem is the same. Memory portability becomes a git clone.

06"Your Harness, Your Memory"

LangChain's deprecation wave was an implicit admission: memory is too coupled to the agent's execution model to be a library feature. The framework can provide primitives (stores, checkpointers, embedding indexes), but the memory architecture must belong to whoever controls the agent loop.

This is an anti-lock-in position. If your memory lives in LangGraph's checkpointers, you can swap the storage backend but not the framework. If your memory lives in files, you can swap everything.

Three Anti-Patterns

01 — Framework-Coupled Memory

Storing memories in a format only one framework can read. LangChain v0.x did this with pickle-serialized ChatMessageHistory. When you migrate, your memories don't come with you.

02 — Invisible Memory

Memory that exists only in a hosted database, inaccessible to the user. The agent "knows" things the developer cannot inspect, audit, or correct. Debugging becomes archaeology.

03 — Undifferentiated Memory

Treating all memory as the same type. A user preference ("dark mode"), a learned procedure ("always run tests before committing"), and a past event ("deploy failed Tuesday") require different storage, retrieval, and update semantics. One vector store does not fit all.

Thesis

The Winning Pattern

Memory should be human-readable (Markdown, JSON, YAML), version-controlled (git), framework-portable (filesystem), and cognitively typed (semantic/procedural/episodic). LangMem gets the types right. The filesystem-as-memory movement gets the portability right. The intersection is where the ecosystem is heading.
