Six-layer memory architecture from product to API

01 / Architecture

Memory Taxonomy

Three groups spanning persistent product features, developer-controlled storage, and ephemeral inference-time mechanisms.

Product-Level
Chat Memory
24-hour synthesis cycle. Model-initiated storage of user preferences, facts, and corrections across conversations.
Chat Search
RAG over conversation history. Retrieves relevant past exchanges at query time.
Managed Agent Sessions
Server-side session persistence for long-running agents. State survives disconnects and reconnects.
Developer-Controlled
Memory Tool
Client-side filesystem tool. API consumers control persistence layer — model reads and writes, host stores.
Claude Code Auto-Memory
Automatic capture of preferences, patterns, and corrections to MEMORY.md between sessions.
CLAUDE.md Hierarchy
Cascading project instructions. Loaded bottom-up from working directory to user root. Developer-authored, git-tracked.
Ephemeral
Extended Thinking
Chain-of-thought scratchpad. Budget-controlled, never persisted, visible only during generation.
Think Tool
Explicit reasoning step within tool-use workflows. Logged in conversation but not persisted cross-session.
Prompt Cache
KV-cache reuse for repeated prefixes. 5-minute or 1-hour TTL. Cost optimization, not semantic memory.
Compaction
Automatic context summarization when conversation exceeds window. Lossy compression of earlier turns.

03 / API

Memory Tool

A beta tool type that gives models persistent read/write memory. The model issues commands; the client executes them against its own storage.

Tool Definition

// Include in tools array
{
  "type": "memory_20250818",
  "name": "memory"
}

Client-side execution model: the API returns memory commands in tool_use blocks. The host application executes them against its own filesystem or database and returns results via tool_result.

Commands

▸ view
+ create
⇄ str_replace
↵ insert
× delete
↺ rename

Execution Flow

// 1. Send message with memory tool enabled
POST /v1/messages { tools: [{ type: "memory_20250818" }] }

// 2. Model returns tool_use with command
{ command: "create", title: "user_prefs", content: "..." }

// 3. Host executes, returns tool_result
{ type: "tool_result", content: "Created user_prefs.md" }

// 4. Model continues with updated context
04 / Patterns

Key Patterns

Context Engineering

Three strategies for managing what the model knows at inference time:

  • Write to context — CLAUDE.md, auto-memory, explicit memory tool writes
  • Select into context — RAG retrieval, chat search, file reads on demand
  • Compress context — compaction, summarization, prompt cache reuse

Progressive Disclosure

Memory layers activate in order of relevance, not all at once:

  • CLAUDE.md loads on session start (always relevant)
  • Auto-memory injects project-scoped context
  • Chat memory surfaces when model detects relevance
  • Full history available via search (on demand)

Cross-Session Learning

Multiple mechanisms accumulate knowledge across conversations:

  • Chat Memory synthesizes every 24 hours from conversations
  • Auto-Memory captures corrections and preferences per-project
  • Memory Tool enables app-specific persistent knowledge
  • CLAUDE.md evolves manually as projects mature
05 / Economics

Prompt Caching

Cache repeated prefixes to reduce latency and cost. Two TTL tiers with different write premiums.

Tier TTL Write Cost Read Cost Net Effect
Ephemeral 5 min 1.25x base input price 0.1x base input price Default. Free auto-refresh on cache hit. Best for interactive sessions.
Extended 1 hr 2x base input price 0.1x base input price Higher write premium but longer retention. Best for batch workloads.
Cache reads are 90% cheaper than uncached input. The write premium pays for itself after 2 cache hits (ephemeral) or 4 hits (extended). Minimum cacheable prefix: 1,024 tokens (Haiku) or 2,048 tokens (Sonnet/Opus).
06 / Key Facts

Status and Milestones

β

Memory Tool: Beta

Type memory_20250818. Client-side execution model. Not yet GA — API surface may change. Requires explicit opt-in via tools array.

✓

Chat Memory: GA

Generally available on claude.ai. 24-hour synthesis cycle. Users can view, edit, and delete stored memories. Model-initiated with user override.

◇

Managed Agents: Apr 2026

Server-side agent sessions with persistent state. Agents survive client disconnects. Scheduled execution and webhook triggers.

↺

Compaction: Automatic

Context window management via lossy summarization. Triggers automatically when conversation length approaches model limits. No developer configuration.

⚙

CLAUDE.md: Convention

Not an API feature but a developer convention adopted by Claude Code. Cascading file hierarchy loaded automatically. Community-driven best practices.

$

Cache Breakpoints

Minimum 1,024 tokens (Haiku) or 2,048 tokens (Sonnet/Opus). Place cache_control at system prompt, tool definitions, or conversation boundaries.
