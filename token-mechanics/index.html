Token Mechanics Explained

How Claude's context window actually works — and why your tools eat most of it.

What Are Tokens?

Tokens are the atomic units of text that language models process. Roughly one token per word, though longer or compound words may split into multiple tokens. Punctuation and whitespace consume tokens too.

That sounds enormous. But the budget covers everything: your messages, Claude's responses, system instructions, and every tool definition loaded into the session.

The backpack analogy

Think of the context window as a backpack with a fixed capacity. The more gear you pack before you leave the house — tool definitions, system prompts, CLAUDE.md files — the less room you have for what you actually pick up on the trail: conversation history, code, analysis.

The Hidden Cost of Tools

MCP tools are integrations that let Claude interact with external services — email, calendars, file systems, APIs. They're powerful. They're also expensive in ways that aren't obvious.

The key insight

Every MCP tool costs tokens on every single message, even if you never use it. Like paying rent on an apartment you never visit — the cost is real whether you show up or not.

Why? Claude needs to know what tools are available before it can decide whether to use them. So every message sends the full tool catalog as part of the prompt.

The math on 460+ tools

Per message: ~3,000–5,000 tokens just on tool definitions

Over 50 messages: ~250,000 tokens burned on tools alone

That's more than the entire 200K budget. Your conversation hits the wall before you've done meaningful work.

Why sub-agents crash

Sub-agents inherit all your configured tools. They start with an already-stuffed backpack, leaving almost no room for the task they were spawned to do. High tool counts cause "Prompt is too long" failures before they even begin.

Controlling Your Tool Load

You don't have to load every tool every time. Two flags give you precise control over what gets loaded into your token budget.

--tools — Choose which services

Each service (Gmail, Calendar, Drive, Sheets, etc.) is a separate module. Without this flag, all services load — up to ~97 tools per account.

--tools gmail,calendar loads only those two services. Everything else stays unloaded.

--tool-tier — Choose depth per service

Within each service, tools are organized into three tiers of increasing capability:

Core
The essentials — search, read, send. 3–7 tools per service.
Extended
Adds admin features — labels, filters, bulk operations. 8–15 more tools.
Complete
Everything, including rarely-used tools. The full catalog.

Tiers are cumulative: "extended" includes everything in "core" plus the extended tools.

CLAUDE.md — Your Project Instructions

CLAUDE.md files are automatic briefing notes. Place one in any folder, and Claude reads it at session start. They define conventions, rules, and context for that directory.

The hierarchy

Claude walks up from your current folder to your home directory, reading every CLAUDE.md it finds. Each level adds context, from broad to specific:

~/.claude/CLAUDE.md Global — role, preferences, universal rules
↓
project/CLAUDE.md Project root — architecture, commands, conventions
↓
project/src/CLAUDE.md Subdirectory — component patterns, test strategies

Keep it lean

Keep each CLAUDE.md under 200 lines. Reference detailed docs instead of embedding them — Claude can always read linked files when it needs more depth. Shorter files mean higher compliance and lower token overhead.

Prompt Caching

Claude Code uses a prompt cache with a 5-minute TTL (time-to-live). The cache stores the system prompt, tool definitions, and recent context — so repeated messages don't re-process all of it from scratch.

Warm Zone
<270s
Cache hit. Faster, cheaper.
→
Danger Zone
270–300s
Cache about to expire.
→
Cold Zone
>300s
Cache miss. Full re-read.

Practical strategy

If you're stepping away briefly, come back within ~4.5 minutes (<270s) to keep the cache warm. That means faster responses and lower cost.

If you're going to be longer than 5 minutes, don't rush back at 4:55 — you'll likely hit a miss anyway. Commit to a longer break (>20 minutes) so the amortized cost of a cold start is negligible.

The worst spot: coming back at exactly 5–6 minutes. You pay for a full cache rebuild but haven't rested long enough to justify it.

The Optimization Spectrum

From quick wins to architectural changes, here's how different approaches compare for reducing tool token overhead.

Approach Token Reduction Complexity How it works
Service filtering --tools 60–80% Low Only load the services you need for this session
Tier filtering --tool-tier ~70% Low Load only core/extended tools per service
Docker profiles Varies Low Swap between "full" and "lean" server sets
mcp-filter proxy ~72% Medium Proxy layer that strips tool definitions before they reach Claude
Gateway lazy-loading ~95% Medium Replace 400+ tools with ~19 meta-tools; load on demand
Start simple

Service filtering (--tools) and tier filtering (--tool-tier core) together give you 60–80% reduction with zero infrastructure changes. Start there. Move to proxy or lazy-loading only when you hit the ceiling.

Quick Reference

Key takeaways

• Every tool costs tokens on every message — even if never invoked. Fewer tools = longer, smarter conversations.

• 200K tokens covers everything: messages, responses, system instructions, and tool definitions combined.

• Use --tools to pick services and --tool-tier core to keep definitions lean. 62% reduction with zero effort.

• Sub-agents inherit all tools. High tool counts cause them to crash before starting work.

• CLAUDE.md files brief Claude automatically — one per folder level, max 200 lines each for 92% rule compliance.

• Prompt cache has a 5-minute TTL. Stay under 270s to keep it warm, or commit to 20+ minutes to amortize a cold start.

• Code CLI and Desktop have separate configs. Changes to one don't affect the other.

• Gateway lazy-loading is the endgame: ~95% token reduction by replacing full catalogs with on-demand meta-tools.
