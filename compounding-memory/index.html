Compounding Memory

A 4-layer architecture for Claude Code — from ghost town vault to semantic memory that makes every session smarter

April 2026 Foundation-first design Karpathy-adapted
01 — The Problem

Four Layers, One Broken

Claude Code has a sophisticated persistence system: four independent memory layers that load context at session start. Three of them work. The fourth — the semantic layer — is a ghost town.

The episodic layers capture 991 observations across 122 sessions. The procedural layer enforces conventions every session. But the vault — intended as the long-term knowledge base — has 46 high-quality entries that never surface automatically. It's a write-only graveyard.

The previous redesign attempt tried to fix this by wiring up retrieval hooks. It failed because it addressed the wrong problem. You can't retrieve what doesn't exist in a retrievable form. The real gaps: no ingestion pipeline, no consistent conventions, no maintenance cycle.

02 — The Insight

Karpathy's LLM Wiki Pattern

In April 2026, Andrej Karpathy published a pattern for LLM-maintained knowledge bases. The core idea: instead of fancy vector databases or RAG pipelines, use the LLM itself to organize knowledge into interlinked markdown files with auto-maintained indexes.

Three-Layer Model
raw/ → wiki/ → schema

Raw sources (immutable inputs) are processed by the LLM into wiki entries (organized, linked, indexed markdown). The schema (CLAUDE.md) defines conventions, structure, and workflows. The LLM handles chunking, relationship-building, and index maintenance.

The key finding: at hundreds to thousands of pages with good indexes, this filesystem approach outperforms vector search. One user turned 383 scattered files into a compact wiki and dropped token usage by 95%.

The scale question, answered: Filesystem with indexes handles hundreds-thousands of documents. Vector search becomes necessary at millions. For a personal knowledge base powering a CLI tool, indexes win on simplicity, cost, and debuggability.
03 — Architecture Decision

Why Filesystem Beats Vector Search

The instinct is to reach for embeddings and similarity search. But at personal-knowledge scale, the tradeoffs don't favor it.

AspectLLM Wiki (Filesystem)Vector RAG
Search method Read indexes, follow wiki-links. Understands relationships because they're explicit. Embedding similarity. Finds "seems related" but misses structured connections.
Infrastructure Markdown files. That's it. Embedding model + vector DB + chunking pipeline.
Maintenance Run a lint. The LLM fixes inconsistencies. Re-embed on every change. Stale embeddings silently degrade.
Cost Tokens only (reading indexes). Ongoing compute + storage + embedding API calls.
Debuggability Open the file. Read the index. Follow the link. Why did similarity return this? Hard to inspect.
Scale ceiling Thousands of pages. Millions of documents.

The decision: index-based retrieval with graph enhancement. An auto-maintained index.md is the primary lookup. Graphthulhu (Obsidian's graph MCP) handles deep traversal for complex queries. If the vault ever exceeds 5,000 entries, revisit.

04 — The Architecture

Four Layers, Four Time Horizons

Each memory layer serves a different time horizon and cognitive function. The key insight: they're not redundant — they're complementary. Like how human memory has working memory, episodic memory, and semantic memory serving different purposes.

The overlap problem, solved: Auto-memory and claude-mem capture similar facts (~70-80% overlap). But they serve different purposes. Auto-memory is the "current state cache" (always loaded). Claude-mem is the "timeline" (searchable history). Neither compounds — that's the wiki's job.
05 — The Convention System

Tags, Relationships, Entry Anatomy

Conventions are the foundation everything else depends on. Without consistent structure, retrieval is guesswork. Without relationships, entries are isolated documents. The schema defines three systems.

Hierarchical Tag Taxonomy

Two closed sets (mandatory, finite) provide the structural skeleton. One open set (grows organically) captures specifics.

domain/tech domain/org domain/personal domain/meta
type/decision type/reference type/pattern type/architecture type/howto type/incident type/profile
memory mcp security nas docker finance agents ...

Every entry gets exactly 1 domain/ + 1 type/ + up to 3 topics. Domain and type drive the directory structure and retrieval logic. Topics grow freely — no approval needed for new ones.

Frontmatter Relationships

Relationships live in YAML frontmatter as arrays of entry filenames. No plugin dependency — Graphthulhu reads these via query_properties.

--- title: Memory Architecture Decisions domain: meta type: decision tags: [memory, architecture] references: [agentic-memory-landscape, agent-memory-architecture] supersedes: [memory-v2-decisions] status: active --- > Four-layer persistence system — v3 active, v4 designing

Start with 4 relationship types. Add more when wiki density demands it:

RelationshipMeaning
referencesCites or draws from another entry
supersedesReplaces an older entry
supportsEvidence that reinforces another entry
relatedGeneral connection (fallback)
06 — Ingestion

Two Paths In, Same Quality Out

The vault has two distinct input channels. Both produce the same quality wiki entries — well-tagged, linked, indexed. The difference is the source, not the output.

Path A — External
raw/ → /ingest → wiki/

Human-sourced content: web clippings, meeting notes (Granola), research articles, NotebookLM exports, manual brain dumps. Dropped into raw/, processed by the /ingest skill into wiki entries.

Path B — Internal
claude-mem → /consolidate → wiki/

Machine-sourced observations auto-captured during sessions. The /consolidate skill queries the SQLite database, clusters observations by topic, and synthesizes them into wiki entries.

The shared principle: Nothing enters wiki/ without LLM organization. Raw observations aren't knowledge — they become knowledge when clustered, synthesized, and linked.

The Processing Pipeline

Content arrives → Extract entities → Check index → Create or Update → Add wiki-links → Update indexes

The LLM decides how to chunk content. A single article might become 5 wiki entries with cross-references. Or it might update 3 existing entries with new information. The index always stays current.

07 — Retrieval

Index-Based Retrieval with Graph Enhancement

At session start, the retrieval hook reads the master index, detects the domain from the working directory, and injects relevant entry summaries into context. Full entries are read on demand.

# SessionStart retrieval flow 1. Read wiki/index.md (~5KB, always) 2. Detect domain from cwd: ~/ClaudeHub/tech/* → read wiki/tech/_index.md ~/ClaudeHub/orgs/* → read wiki/orgs/_index.md ~/vault/* → read wiki/meta/_index.md 3. Match SPEC.md keywords against entries (top 3-5) 4. Inject entry summaries (~2-4KB) 5. On-demand: Graphthulhu for deep reads (when needed)

08 — Build Phases

Foundation Before Hooks

The critical ordering: conventions must exist before ingestion can produce well-formed entries. Entries must exist before retrieval has something to surface. Consolidation and maintenance come last — they improve a working system, not build one.

Phase 0a-c: Convention System Complete
Tag taxonomy, relationship vocabulary, entry anatomy, vault/CLAUDE.md wiki schema. Graphthulhu audit confirmed frontmatter-only relationships work.
Phase 0d: Migration Next
Create wiki/ directory structure. Migrate 46 existing entries with convention-compliant frontmatter. Build master index.md and domain sub-indexes.
Phase 1: Ingestion Pipeline 1-2 sessions
Build /ingest skill. Process raw/ inputs into wiki entries. Test with existing clippings and meeting notes.
Phase 2: Retrieval Hook 1 session
SessionStart vault injection. Domain detection from working directory. Context-matched summary injection.
Phase 3: Scored Injection 1 session
Replace flat recency with ADR-002 weighted scoring. Enable semantic search. Document authority model.
Phase 4: Consolidation + Maintenance 1-2 sessions
Build /consolidate (claude-mem → wiki). Enhance /kb-audit with Karpathy-style lint. Integrate Graphthulhu health checks.
The end state: Every session's learnings feed into wiki entries that improve future sessions. Knowledge compounds like interest — each session makes the next one smarter.
